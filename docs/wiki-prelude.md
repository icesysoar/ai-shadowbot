## 2. 对本项目的避坑建议

### 2.1 截图识别误差（最易被低估的风险）
- **不要纯靠 OCR/视觉识别定位**。历史经验中"元素定位 + 置信度阈值"远胜纯截图比对。
- 建议**双轨定位**：① 元素/控件句柄锚定（优先）；② 截图模板 + 坐标兜底。两者均记录**置信度**，低于阈值触发人工确认而非盲猜。
- 建立**元素指纹库**（控件特征哈希），定期重标定，应对窗口缩放/DPI/主题变化。
- 执行前**回读屏幕状态**确认上一步真实生效（呼应 2.4 "生成 ≠ 执行"）。

### 2.2 权限申请（沙箱/进程/UAC）
- 复用 `taskkill 被沙箱拦截` 教训：涉及进程管理、文件写入、注册表的动作**必须走授权通道**，不可假设裸命令可用。
- 敏感操作（删文件、改系统设置、外发网络）默认**需用户显式授权**；应用作用域用**白名单**限定（哪些窗口/进程可被操作）。
- 遵循**权限最小化**：非必要不请求管理员/UAC，避免体验与安全隐患。

### 2.3 跨平台兼容
- **编码**：所有 Python 脚本首行加 `sys.stdout.reconfigure(encoding='utf-8', errors='replace')`（来自 GBK 崩溃避坑），杜绝中文/emoji 日志崩溃。
- **快捷键/坐标差异**：macOS 与 Windows 的修饰键、菜单结构不同；优先用**语义动作**（"点确定"）而非硬坐标，坐标仅作兜底且需 DPI 归一。
- **分辨率/DPI**：截图模板按逻辑像素归一，避免 125%/150% 缩放下失配。

### 2.4 LLM 输出动作的安全校验（核心风险，直接复用沙箱思想）
- 把 LLM 产出的"动作序列"当作**待校验代码**处理，套用 `Python 沙箱黑名单模式`：
  - 用**黑名单**而非白名单——禁止 `rm -rf`、系统级命令、`open` 受保护路径、外发网络等危险原语；其余正常放行，避免可用性灾难。
  - 危险原语：`format`, `eval`, `exec`, `__import__`, 直接 shell 执行、写 `C:\Windows`/`/etc`、未授权 `curl`/`requests` 外联。
- **执行前**：dry-run 静态校验 + 危险动作二次确认弹窗。
- **执行后**：**回读屏幕/系统状态确认动作真实生效**（直接迁移 `妙想 API 提交 ≠ 成交` 原则——"模型说做完"≠"真做完"）。
- 加**超时与步数上限**（`multiprocessing.Process(timeout=N)` + 最大步数熔断），防止 Agent 跑飞。

### 2.5 Agent 循环可靠性（迁移 Loop Engineering）
- 明确**停止标准与熔断**：连续失败 N 步 / 超时 / 无法定位目标 → 中止并 report，而非无限重试。
- 维护**上下文状态**（当前窗口/已执行动作/中间结果），支持断点续跑。

### 2.6 录制/回放控制面板 UI（迁移前端 JS 避坑 + esc）
- 事件监听用 `_bound` 标志防重复绑定；静态资源加 `?v=` 破缓存；`.catch` 必带 `console.error`；用户数据渲染一律 `esc()`。

## 4. 结论

- **无同构历史项目**可直接照搬，但现有 Wiki 的"安全沙箱 / 原子动作 / 循环架构 / 跨平台 / UI 安全"集群**高度契合** AI 影刀的四大风险面。
- 建议在项目启动前，将上述 8 条概念作为**种子知识**注入 `自动系统/wiki`，使后续 ReqAnaly 规划与 AutoCode 实现能经 `wiki_query` 自动召回避坑经验。
- 优先级最高：**LLM 动作黑名单安全校验** 与 **动作生成 ≠ 动作执行**——二者直接决定产品是否"可用且安全"。
