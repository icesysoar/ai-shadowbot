## 1. 全面差距对比矩阵

### 1.1 核心能力矩阵

| 维度 | 影刀 (Yingdao RPA) | ai-shadowbot (当前) | 差距 |
|------|-------------------|-------------------|------|
| **动作组件数量** | **400+** 预置组件（网页/Excel/邮件/DB/文件/OCR/AI/桌面/系统/图像/手机等） | **9** 个原子动作（click/type/key_press/screenshot/wait/open_app/scroll/double_click/right_click） | 🟠 差390+ |
| **可视化流程编辑器** | ✅ 拖拽式积木搭建，左侧指令区→中央画布→右侧属性面板 | ❌ 纯 CLI 聊天界面，无图形画布 | 🔴 重大缺口 |
| **流程编排能力** | 条件判断(if/else)、循环(for/while)、分支(try/catch)、子流程调用、等待 | 仅 LLM 生成的线性动作序列，无结构化流程控制 | 🔴 重大缺口 |
| **变量系统** | 全局变量(glv)、局部变量、变量管理面板、类型转换 | ❌ 无变量系统，动作间数据传递靠硬编码 | 🔴 重大缺口 |
| **触发器** | 定时触发、热键触发、文件变化触发、邮件到达触发、手动触发 | ❌ 仅 CLI 手动触发 | 🔴 重大缺口 |
| **错误处理** | try/catch、自动重试(可配次数+间隔)、跳过、终止、降级 | ❌ 线性执行，无错误恢复，失败即中断 | 🔴 重大缺口 |
| **调试模式** | 单步执行、断点、回放速度调节、变量监视 | ❌ 仅 dry_run 看动作序列 | 🟠 差得远 |
| **调度执行** | 可视化调度台、定时任务、循环执行、日历配置 | ❌ 无调度 | 🔴 重大缺口 |
| **日志回溯** | 执行日志、运行记录、失败截图、全量审计 | ⚠️ 仅 SessionRecorder 基本记录 | 🟠 差得远 |
| **AI 能力** | 魔法指令 3.0（自然语言→搭流程）、AI 操作屏幕、智能修复 | ✅ LLM 规划 + 视觉坐标 + 安全护栏 | 🟡 方向对但深度浅 |
| **安全护栏** | 基础应用权限控制 | ✅ 强安全（deny-unconfirmed / AC4/AC5/AC6 / ESC急停） | 🟢 领先 |
| **跨平台** | Windows / Mac / Android / 信创 | ⚠️ 仅 Windows (PyAutoGUI) | 🟠 差得多 |
| **生态** | 影刀学院、应用市场、1000+场景库、企业版管理台 | ❌ 无生态 | 🔴 无 |
| **L5 外部 AI 触发** | ❌ 封闭生态 | ✅ soara-l5bridge（HTTP+MCP 外触发） | 🟢 领先 |
| **开源/可自控** | ❌ 商业闭源 | ✅ 全 Python 开源，可自控修改 | 🟢 领先 |

|------|------|------|

## 2. 核心差异：不是"做一样"，而是"做得更好"

### 影刀的模式
```
用户手动拖拽节点 → 保存流程 → 触发执行 → 拿结果
              ↑
        全靠人手搭，门槛低但有上限
```

### 用户要的模式（"AI代替人搭节点"）
```
AI 理解需求 → AI 自动编排节点 → 用户审查确认 → 执行 → 结果回流
                      ↑
              或: 用户拖拽节点
                  AI 辅助补全/优化
```

**核心创新点**：影刀需要人一个个拖节点。我们要的是**AI 自主搭建工作流（可视化节点）→ 用户可审查/修改 → 自动执行**。

## 3. 新架构方案：AI 驱动的工作流引擎

### 3.1 架构总览

```
┌─────────────────────────────────────────────────────┐
│                   用户入口层                          │
│  CLI聊天  │  Web画布(拖拽)  │  L5外部AI  │  定时器   │
└──────────────────┬──────────────────────────────────┘
                   ▼
┌─────────────────────────────────────────────────────┐
│               AI 工作流编排层 (核心创新)              │
│  ┌─── 自然语言→工作流编译器 ───┐                     │
│  │   "帮我每天9点查邮件并整理Excel"                    │
│  │   → 自动生成节点流(可预览/修改)                    │
│  └────────────────────────────┘                     │
│  ┌─── 工作流执行引擎 ──────────┐                     │
│  │   条件/循环/分支/变量/重试   │                     │
│  └────────────────────────────┘                     │
└──────────────────┬──────────────────────────────────┘
                   ▼
┌─────────────────────────────────────────────────────┐
│            动作执行层 (现有 F002 + F005)              │
│  鼠标/键盘   │  截图+视觉   │  终端命令   │  安全护栏  │
└─────────────────────────────────────────────────────┘
```

### 3.2 核心组件定义

| 组件 | 说明 | 对标影刀 |
|------|------|---------|
| **工作流编译器** | LLM 把自然语言需求编译成结构化节点图（JSON/DAG） | 对标影刀"魔法指令"但更强 |
| **节点调色板** | 可扩展的动作节点库（现有 9 个 + 逐步增加） | 对标影刀指令库 |
| **可视化画布** | Web 界面的拖拽节点编辑器 | 对标影刀积木搭建 |
| **工作流引擎** | 解释执行节点图（条件/循环/分支/变量） | 对标影刀执行引擎 |
| **变量系统** | 节点间数据传递 | 对标影刀全局变量 |
| **错误策略** | 重试/跳过/终止/降级策略配置 | 对标影刀错误处理 |
| **调度器** | 定时/热键/事件触发 | 对标影刀调度台 |
| **执行日志** | 节点级执行追踪、失败回溯 | 对标影刀日志 |

## 4. 分阶段实现路线图

### Phase 1（MVP 2-3周）— 先跑通"AI 编译工作流 + 执行"

| 模块 | 内容 | 说明 |
|------|------|------|
| 工作流数据结构 | DAG 节点图 JSON schema | 复用现有 actions.py 动作类型 |
| AI 工作流编译器 | 自然语言 → DAG 节点图 | 基于现有 planner 扩展 |
| 工作流执行引擎 | DAG 解释执行器（线性+条件+循环） | 新增 runtime/engine.py |
| 变量系统 | 基础变量作用域（节点间传参） | 新增 runtime/variables.py |
| 错误策略 | 重试/跳过/终止 | 新增 runtime/errors.py |
| CLI 入口 | `--run-workflow`、`--compile` | 扩展 cli.py |

**交付**：CLI 模式下，用户说"帮我每天9点查邮件整理Excel"→ AI 自动生成节点流→ 展示→ 确认执行

### Phase 2（+2-3周）— 可视化画布

| 模块 | 内容 |
|------|------|
| Web 画布前端 | React Flow / LiteGraph.js 拖拽节点编辑器 |
| Python 后端 API | FastAPI 提供 workflow CRUD + 执行入口 |
| 节点调色板 | 分类展示可用动作节点，支持搜索 |
| 流程持久化 | SQLite 存储工作流定义 |
| 用户审查模式 | AI 生成的流可手动拖拽修改 |

**交付**：Web 界面 + AI 自动搭节点 → 用户审查拖拽修改 → 执行

### Phase 3（+2周）— 调度与生态

| 模块 | 内容 |
|------|------|
| 定时调度器 | cron 触发 + 热键触发 |
| 执行日志回溯 | 节点级日志 + 失败截图 |
| 流程模板市场 | 保存/分享/复用工作流 |
| L5 bridge 集成 | 外部 AI 可触发工作流 |

### Phase 4（长期）— 超越影刀

| 模块 | 内容 |
|------|------|
| 智能错误修复 | 执行出错→AI 分析原因→自动修正节点 |
| 流程优化建议 | 根据执行数据分析瓶颈→建议优化节点顺序 |
| 多 Agent 并行 | 复杂流程拆分子工作流→并行执行 |
| 跨平台 | macOS/Linux 支持 |

## 5. 与 L5 bridge 的集成

```
┌─────────────┐    HTTP/MCP     ┌──────────────────┐
│ trae / 网页AI│ ──────────────→ │  soara-l5bridge   │
│ 其他AI Agent │                 │  (触发网关)       │
└─────────────┘                 └────────┬─────────┘
                                         │ POST /l5/start
                                         ▼
                               ┌──────────────────┐
                               │ 工作流编译器      │
                               │ (AI 编译+执行)    │
                               └──────────────────┘
```

L5 bridge 让外部 AI 也能触发工作流编译和执行。这样：
- trae 发需求 → L5 bridge → 编译工作流 → 执行 → 结果返回
- 网页 AI 聊天界面 → HTTP 调用 → 同上
- 定时器 → 内部触发 → 同上

## 7. 关键决策点（待主理人拍板）

1. **画布方案**：嵌在 CLI 里用浏览器打开（FastAPI 本地 serve），还是独立 Web 应用？
2. **工作流编译器粒度**：AI 全自动编（用户只审查确认），还是 AI 编骨架+用户拖拽调细节？
3. **节点数起步**：先只打包现有 9 个动作，还是 Phase 1 就扩一些关键组件（如截图OCR、网页表单）？
4. **持久化**：工作流定义存 SQLite（轻量）还是 JSON 文件？
