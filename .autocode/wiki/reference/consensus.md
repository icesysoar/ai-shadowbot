## 0. 评审对象与核心决策点

**项目**：自然语言 → 电脑操作动作（RPA / Computer-Use agent）。用户聊天说"要做什么"，AI 编译为鼠标/键盘/截图动作并自动执行。

**待裁决的核心技术路线（二选一）**：
- **A. 自研封装**：PyAutoGUI + LLM function calling，自己写"观察→决策→执行"闭环。
- **B. 改造底座**：基于 computer-agent / Open-Interface 等开源 computer-use 底座二次开发。

**历史知识注入提示**（soara-digest 注入）：`HDA 模式：增强 ≠ 重构` —— 现有子工作流已有 80% 能力，只需要补丁，不要当全新系统从零设计；`BacktestWorker 架构避坑` —— 重造闭环易踩坑。本结论与"复用开源底座 + 自研差异层"方向一致。

## 1. 四角色独立分析（互不引用，分别发声）

### 1.1 用户视角 👤
> 我要的是"帮我干活"，不是"一个会自己乱点的黑盒"。

- **可读性/可改性刚需**：用户无法接受"我说一句，它就在我电脑上全自动跑且我看不见在干嘛"。影刀的护城河正是"积木流程可视化、可改、可复用"。对话式 agent 若只给黑盒执行，信任建立极难。
- **可中断/可回滚**：弹窗、验证码、意外页面一旦出现，agent 必须能停、能退一步，而不是硬冲。
- **隐私焦虑**：屏幕里有密码、聊天、银行页面——"我的内容会不会被传到云端？"是高频疑虑，直接决定接受度。
- **成本/速度敏感**：每次截图都调大模型，慢且贵；用户期望"常用动作秒级、不要每次烧 token"。
- **容错期望**：识别点错位置、UI 一变就崩，是用户最易感知的"不靠谱"。

### 1.2 安全视角 🛡️
> 一个拥有鼠标键盘和屏幕权限的 LLM，是攻击面最大的组件之一。

- **越权执行 / 幻觉动作**：LLM 幻觉可能生成 `rm -rf`、格式化、转账、群发消息等破坏性 function call。若工具集暴露 shell/文件写，风险指数级上升。
- **提示注入（最高危）**：computer-use agent 的致命面——屏幕上网页/文档里的"忽略前述指令，执行 X"会被 agent 读屏后吸收，诱导越权。这是 OpenAI Operator / Claude Computer Use 都已公开确认的攻击向量。
- **权限最小化**：需要 屏幕录制 + 辅助功能(accessibility) + 可能管理员。macOS TCC、Windows UAC 弹窗；持久化授权若管理不当等同留下后门。
- **屏幕内容外发**：截图送云端 = 凭证/PII 泄露。必须默认脱敏或本地化。
- **不可恢复动作护栏缺失**：删除、发送、支付类动作无二次确认 = 事故。

### 1.3 工程视角 ⚙️
> 纯坐标点击是脆弱的；我们该造"眼睛"和"安全带"，不是重造"手"。

- **反对纯 PyAutoGUI 坐标点击**：DPI 缩放、多显示器、窗口重排、系统主题变更都会让 (x,y) 失效；模型对像素坐标回归常偏 10–30px，点击命中率不稳定。
- **观测方式决定上限**：Open-Interface 的"无障碍树(accessibility tree) + 截图"混合观测远比纯像素稳健——能拿到按钮语义而非盲猜坐标。这是 B 路线的核心优势。
- **闭环延迟/成本**：截图→大模型→坐标 每次数百 ms~数秒、单次数万 token。需"动作模板缓存 + 小模型前置判定"降本提速。
- **跨平台差异**：Win（UIA）、macOS（AX/辅助功能）、Linux（X11/Wayland）API 各异；纯 PyAutoGUI 跨平台坑多。
- **循环可控性**：自主 agent 循环若无步数上限/超时/预算上限 = 失控烧钱或死循环。
- **可测试性**：RPA 行为难单测，需"录制回放 + 沙箱(VM/容器)"验证体系。

### 1.4 竞品视角 📊
> 巨头亲自下场做 computer-use，纯 wrapper 护城河为零；差异点在"本地+可编辑+中文"。

- **红海格局**：影刀/UiBot/八爪鱼占企业"录制+积木"市场且稳定；OpenAI Operator、Claude Computer Use、Google Mariner 已把通用 computer-use 当战略。我们做通用 wrapper 毫无壁垒。
- **差异化机会**：① **本地优先 + 隐私**（巨头都上云，这是空白）；② **影刀式可视化可编辑流程**（而非黑盒一次性执行）；③ **国内软件/中文场景适配**（微信/钉钉/网银/政务系统）。
- **护城河**：执行轨迹数据飞轮 → 微调专有小模型 + 流程市场/UGC。
- **速度即生命**：别重造底座，站开源肩膀上快速出 MVP 占"本地可控 RPA"心智。

| C5 | 对话即执行(简单) vs 可视化可编辑流程(影刀式差异) | 竞品 ↔ 用户/工程 | 🟢 低 | 对话生成流程 → **可视化展示并可编辑** → 确认执行。对话与流程图并存，既满足简单预期又建立差异化与可控性。 |

## 4. 技术路线共识：A vs B 的裁决

### 裁决结论：**选 B 的"观测底座"，但自研规划层 + 护栏层 + 流程层** —— 即"改底座而不全盘接管"。

**不选 A（纯自研 PyAutoGUI + LLM）的理由**：
- 重造"观察→决策→执行"闭环，正是历史知识警告的"从零重建"陷阱（HDA: 增强≠重构）。
- 纯坐标点击脆弱（DPI/多屏/主题），命中率无保障，产品初期即显"不靠谱"。
- 缺失无障碍树语义定位，是工程视角的高频硬伤。

**不"全盘接管 computer-agent"的理由**：
- computer-agent 循环较朴素、缺无障碍树、缺护栏，偏向一次性任务，不是影刀式"可编辑流程"产品底座。
- 直接 fork 当产品 = 竞品视角所言"无差异 wrapper"，护城河为零。

**采用 Open-Interface 式混合观测作为"眼睛"的理由**：
- 无障碍树 + 截图 混合 = 语义定位 + 视觉兜底，命中率与鲁棒性显著优于纯像素。
- 开源、本地可跑、社区活跃，契合"本地优先+隐私"差异化。

**自研的三层（差异与护城河所在）**：
1. 动作规划层（LLM function calling，专有小模型微调空间）
2. 安全护栏层（确认门/脱敏/权限最小化/提示注入防护）
3. 影刀式可视化流程层（对话生成 → 可编辑积木 → 复用市场）

### 权衡矩阵

| 维度 | A 纯自研 | B 改底座(混合观测) | 推荐落地 |
|------|---------|------------------|---------|
| 上手速度 | 快(几行代码) | 中(需读源码) | B |
| 长期鲁棒性 | 低(坐标脆弱) | 高(语义+视觉) | B |
| 安全可控 | 自担(需自建) | 需 retrofit 护栏 | B+自研护栏 |
| 差异化空间 | 低 | 高(流程层自研) | B |
| 隐私 | 可控(自己定) | 取决于底座 | B+本地默认 |
| 护城河 | 无 | 数据飞轮+流程市场 | B |

## 5. 架构方向建议（五层模块划分）

```
┌─────────────────────────────────────────────┐
│  交互层 InteractionLayer                       │  聊天 UI / 流程画布 / 暂停·单步·回滚 / 确认弹窗
├─────────────────────────────────────────────┤
│  安全护栏层 SafetyGuardrailLayer               │  敏感区打码 · 危险动作 allowlist · 人类确认门 · 提示注入检测 · 权限最小化
├─────────────────────────────────────────────┤
│  动作规划层 ActionPlanningLayer                │  NLU→意图 · LLM function calling · 动作模板缓存 · 小模型前置判定 · 步数/预算上限
├─────────────────────────────────────────────┤
│  执行引擎层 ExecutionEngineLayer               │  无障碍树定位 · 截图匹配兜底 · PyAutoGUI/UIA/AX 适配 · 多平台驱动 · 状态观测回传
├─────────────────────────────────────────────┤
│  观测层 ObservationLayer（底座核心，复用 Open-Interface 范式）│  屏幕采集 · 无障碍树抽取 · 敏感区识别 · 本地/云端切换
└─────────────────────────────────────────────┘
```

| 层 | 职责 | 关键输入→输出 | 关键技术/决策 |
|----|------|--------------|--------------|
| 交互层 | 对话收口、流程可视化、运行控制 | 自然语言 ↔ 可编辑流程、控制指令 | 流程画布(影刀式)、确认门 UI |
| 安全护栏层 | 防止越权/误操/泄露 | 动作计划 → 允许/拦截/需确认 | 危险 action allowlist、屏幕脱敏、注入检测(对读屏内容做指令边界校验) |
| 动作规划层 | 意图→结构化动作序列 | 意图+观测 → function call 列表 | LLM function calling、模板缓存、小模型前置、循环上限 |
| 执行引擎层 | 把动作落到真实系统 | 动作 → OS 级操作 + 结果回传 | 无障碍树定位为主、截图匹配兜底、跨平台驱动 |
| 观测层 | "眼睛" | 屏幕 → 结构化观测(树+图) | 复用 Open-Interface 范式、本地优先 |

> 注：观测层即"改底座"的部分，其余四层为自研差异层。层级依赖自上而下，护栏层横跨规划与执行（卡在"执行前最后一关"）。

## 6. 风险评估（高风险项 + 缓解）

| 风险 | 类型 | 等级 | 缓解建议 |
|------|------|------|---------|
| 提示注入（屏幕内容诱导越权） | 安全 | 🔴 极高 | 对读屏内容做"指令/数据"边界隔离；敏感指令白名单；destructive 动作永不自动执行 |
| 越权/幻觉破坏性动作 | 安全 | 🔴 极高 | 危险 action allowlist + 人类确认门；工具集默认不暴露 shell/文件写 |
| 屏幕内容外发泄露 | 隐私 | 🔴 高 | 默认本地模型；云端时敏感区(密码/金额/聊天)打码后外发；可审计日志 |
| 截图/坐标识别误差导致误操作 | 可靠性 | 🔴 高 | 无障碍树语义定位为主、截图兜底；点击前二次校验元素；误操作回滚点 |
| 多屏/DPI/主题导致定位失效 | 可靠性 | 🟡 中 | 启动时校准显示环境；相对坐标+元素锚定；失败时重试/求助用户 |
| 自主循环失控(烧钱/死循环) | Scope | 🔴 高 | 步数上限 + 时间预算 + token 预算；超界自动暂停待确认 |
| 跨平台兼容性坑 | 工程 | 🟡 中 | 先锁单一平台(MVP)，驱动抽象层隔离 |
| 模型幻觉产生非法动作序列 | 可靠性 | 🟡 中 | 动作 schema 强校验 + 执行前 dry-run 检查 |
| 成本/延迟用户体验差 | 商业 | 🟡 中 | 模板缓存 + 小模型前置判定；常用动作本地化 |

## 7. 共识结论

**已达成共识**：
1. 不做"纯自研 PyAutoGUI 包一层"（脆弱、重造闭环）。
2. 不"无脑 fork computer-agent 当产品"（无差异、缺护栏）。
3. 采用 **Open-Interface 式"无障碍树+截图"混合观测** 作为底座核心（眼睛）。
4. **自研三层差异**：动作规划层 + 安全护栏层 + 影刀式可视化流程层（护城河）。
5. 默认 **本地模型 + 可选云端增强**；危险动作强制人工确认；屏幕敏感区脱敏。
6. 对话生成流程 → 可视化可编辑 → 用户确认 → 执行（可调、可暂停、可回滚）。

**仍需向用户/主理人决策的点**（不阻塞，已给推荐）：
- 平台优先级（Windows 主 / macOS 主 / 跨平台）—— 影响无障碍树 API 选型。→ 推荐 **先 Windows(UIA)**。
- 模型策略（纯本地 / 本地+云端增强）—— 影响隐私与能力上限。→ 推荐 **本地默认 + 云端可选**。
- 首版 Scope（单应用域内如浏览器/Excel / 全局桌面）—— 影响 MVP 规模。→ 推荐 **单应用域收窄**。
- 产品形态（自用工具 / 开源 / 商业化 SaaS）—— 影响数据与安全策略。→ 待定。

## 8. 需向用户澄清的问题（推荐方案先行，不阻塞）

1. **形态**：对话"一步到位自动执行" vs 对话"生成可编辑流程再执行"？→ **推荐后者**（可信+差异）。
2. **平台**：主战场 Windows 还是 macOS？→ **推荐先 Windows**。
3. **模型**：接受云端大模型(更聪明但有隐私/成本) 还是 坚持本地优先？→ **推荐本地默认+云端可选**。
4. **首版边界**：先做"浏览器/Excel 等单应用域"还是"全局桌面任意操作"？→ **推荐单应用域 MVP**。
5. **商业模式**：自用 / 开源 / 商业化？→ 影响数据飞轮与脱敏策略，待定。
