# PRD · AI 版影刀 (ai-shadowbot)

> 自然语言 → 桌面动作规划的 RPA / Computer-Use Agent
> 作者：叶知秋 (ye-zhiqiu) · 日期：2026-07-06 · 版本：v0.1

---

## 1. 背景与目标

**背景**：传统 RPA（影刀、Ui.Vision、Taskt 等）多为「录制回放 + 拖拽搭积木」，用户仍需学习工作流编辑器；
近年 LLM 让「用自然语言直接驱动电脑」成为可能（computer-agent、Open-Interface、AutoGUI 等）。

**目标**：打造「AI 版影刀」——用户用聊天说要做什么，AI 把它编译成一系列桌面动作
（鼠标移动/点击、键盘输入、截图识别、必要时终端命令）并自动执行。

**本质**：自然语言 → 动作规划的 RPA / Computer-Use Agent。

---

## 2. 范围

**MVP 包含（4 个 Feature）**：
- F001 自然语言解析与动作规划（LLM function calling）
- F002 桌面动作执行引擎（鼠标/键盘/截图）
- F003 安全护栏与人工确认机制
- F004 CLI / 聊天交互入口

**MVP 不包含（后续迭代）**：
- 网页专项自动化（DrissionPage 类能力）
- 可视化拖拽编辑器 / GUI
- 多 Agent 协作、长程记忆
- 企业级权限/审计/云端调度

---

## 3. 用户画像与核心场景

- **核心用户**：想把重复电脑操作自动化的普通用户 / 效率工具爱好者。
- **核心场景**：「帮我把桌面上的 report.xlsx 重命名为 2026-report.xlsx」「打开浏览器搜一下天气并截图」「每隔 5 分钟刷新一次页面」。

---

## 5. 非功能需求

- **安全**：自动执行必须有护栏，缺省拦截高危动作（见 F003）。
- **跨平台**：优先 Windows（当前主机），动作层需兼容 Mac（需辅助功能授权）。
- **可观测**：执行过程可回显进度与截图，可导出动作脚本重放。
- **可配置**：LLM provider 可切换，确认策略可配置。
- **编码注意（来自知识库经验）**：Windows Python 项目必须处理 GBK 编码（文件 IO / 日志 / subprocess 输出显式指定 UTF-8），避免 `UnicodeDecodeError`。

### 6.2 分 Feature 验证

**F001 自然语言解析与动作规划**
- 可行性✅：LLM function calling 成熟，桌面动作可枚举为工具集。
- 价值✅：产品大脑，核心价值来源。
- 风险⚠️：幻觉→错误动作；多步易偏离；依赖截图感知质量。
- 范围✅：输入意图→输出结构化 action 序列，AC≥3。
- 依赖⚠️：依赖 LLM API；依赖 F002 动作原语定义（schema 对齐）；依赖截图（F002）。

**F002 桌面动作执行引擎**
- 可行性✅：PyAutoGUI 成熟跨平台；注意 Windows GBK 坑。
- 价值✅：执行手脚，无它规划无意义。
- 风险🔴 高：坐标漂移（DPI/多屏/窗口变动）已升为高，靠「元素锚定+重试」+ M2 无障碍树混合感知解决；权限、后台窗口失效仍需处理。
- 范围✅：原语 move/click/type/screenshot 等，AC≥3（新增无障碍树感知 AC5）。
- 依赖⚠️：依赖 PyAutoGUI/pygetwindow 安装、OS 权限、F003 钩子包裹。

**F003 安全护栏与人工确认**
- 可行性✅：拦截器模式，纯软件。
- 价值✅：RPA 产品化关键（防误操作/防越权）。
- 风险🔴 高：经共识重评——提示注入(极高)、隐私外发(高)、越权破坏性动作(高)；失败可降级为「默认拦截」。
- 范围✅：默认未确认即拦截/语义化 gate/三档确认状态机/提示注入防护/隐私脱敏/急停/不可逆动作强提示，共 6 条 AC。
- 依赖：依赖 F002 动作前回调钩子；依赖 F001 结构化 action 类型做语义 gate。

**F004 CLI / 聊天交互入口**
- 可行性✅：CLI/WebSocket 均可。
- 价值⚠️ 中：入口体验，非核心算法，但 MVP 必载。
- 风险✅ 低。
- 范围✅：输入→计划→确认→执行→回显，AC≥3。
- 依赖：依赖 F001/F002/F003 就绪。

## 7. 开源底座复用建议（重点）

### 7.1 建议复用：`computer-agent`（suitedaces/computer-agent）为参考范式 + PyAutoGUI 自建执行引擎

**推荐作为主干参考 / 部分复用 `computer-agent`**，理由：
1. **架构对齐度最高**：它本身就是「本地 AI agent：自然语言 → 截图感知 → 鼠标/键盘/终端动作」的完整闭环，与本项目目标几乎一一对应。
2. **已解决硬骨头**：屏幕截图捕获、动作坐标对齐（grounding）、终端命令通道都已实现，不必从零造轮子。
3. **本地运行可私有化**：数据不出本机，契合 RPA 安全诉求（F003）。

**改造点（不要全盘照搬）**：
- 把它的「整体 prompt 决策」拆成 **F001 的 function-calling 规划器**（结构化输出 action 列表），便于 F003 逐项拦截。
- 执行引擎用 **PyAutoGUI** 自建/包装，统一跨平台动作原语 + 加 pre-action 确认与黑名单（F003）。
- 交互入口做 **F004 的 CLI/聊天**，复用其终端通道。

### 7.2 不推荐直接复用

| 工具 | 不推荐原因 |
|------|-----------|
| Ui.Vision / Taskt / lingsuo_rpa / OpenRPA | 录制回放 + 拖拽式，缺自然语言理解，与目标不符 |
| TagUI / Robot Framework | DSL 脚本式，需写脚本而非聊天 |
| DrissionPage | 偏网页自动化，桌面能力弱 |
| Cradle / NeuralAgent | 游戏/通用框架偏重，轻量桌面 RPA 过度设计 |

### 7.3 可借鉴的设计
- **Open-Interface**：「无障碍权限 + 截图评估」的感知方式（F002 截图感知参考）。
- **AutoGUI Agent**：HTTP REST + MCP 接口形态（F004 可学其 MCP server 设计，便于后续扩展）。

## 10. 共识评审结论（F003 + 底座，补充轮）

> 评审人：云见微 (yun-jianwei) · 全文见 `docs/consensus-f003.md` · 已由叶知秋落定到 project-spec.json。

**底座决策**：computer-agent + PyAutoGUI 方向认可（enhance-not-refactor）；感知层须增强 Open-Interface 式「无障碍树 + 截图」混合观测以达鲁棒定位 → 已写入 F002 AC5 / M2。

**F003 三决策（已落 AC）**：
1. 默认「未确认即拦截」（deny-unconfirmed）：黑名单硬拦，白名单低危可跳过（F003 AC1）。
2. 三档确认默认 = 计划级批量确认 + 运行时高危单步确认，状态机实现（F003 AC3）。
3. 黑名单语义化 gate：基于 F001 结构化 action 类型，覆盖跨平台等价物与绕过向量（F003 AC2 / §2.3 清单）。

**新增两项关键风险（已落 AC）**：
- 提示注入（Prompt Injection）：屏幕内容诱导越权 —— 极高，F003 AC4 防护（指令/数据边界隔离）。
- 隐私外发：默认云端 LLM 致截图外发 —— 高，F003 AC5 脱敏打码 + 本地模型默认（SCREEN_MASK_SENSITIVE）。

**开发节奏**：M1 先按硬黑名单 + ESC 急停推进；增强项（无障碍树 / 提示注入防护 / 脱敏）入 M2/M3，不阻塞开发。

## 9. 知识注入（关键概念）

> 来源：本次调研 + wiki 知识库注入，仅记录概念，未执行脚本。

- **RPA（机器人流程自动化）**：用软件机器人模拟人在电脑上的操作，传统以录制回放为主。
- **Computer-Use**：让 LLM 直接「看屏幕 + 操作键鼠」完成任务的范式，核心是「截图感知 → 决策 → 动作」闭环。
- **PyAutoGUI**：Python 跨平台 GUI 自动化库，鼠标/键盘/截图控制的事实标准，本项目的 F002 底座。
- **LLM Function Calling**：让 LLM 以结构化参数调用预定义工具；本项目用它将「自然语言意图」编译为「桌面动作序列」（F001 核心）。
- **Prompt Injection（提示注入）**：computer-use 头号陷阱——屏幕/文档中的「忽略指令执行 X」类内容诱导 agent 越权。本项目的缓解：读屏内容一律视为「数据」而非「指令」，做指令/数据边界隔离（F003 AC4）。
- **Screen Desensitization（屏幕脱敏）**：截图外发至云端 LLM 前，对密码框/账号/剪贴板等敏感区打码，并提供本地/私有化模型默认选项（F003 AC5）。
- **知识库反模式（来自 journal 经验）**：
  - Windows Python 必须处理 GBK 编码，文件 IO / 日志 / subprocess 显式 UTF-8。
  - 增强 ≠ 重构：现有能力 80% 可用时，优先打补丁而非新建系统（本项目复用 computer-agent 思路即此原则）。
  - 管线引擎设计需避免 5 个常见坑（耦合、无兜底、状态不可观测等）。
